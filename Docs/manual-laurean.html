Capítulo 01

Arquitectura real del sistema

Laurean Shop es un sitio estático (HTML, CSS y JavaScript sin framework de frontend) con Supabase como backend en la nube y deploy en Vercel. No requiere instalar nada en el equipo del usuario más allá de un navegador moderno.

Las páginas del sistema

Tienda pública (Laurean.html)

Catálogo, carrito y checkout para clientes finales. Accesible desde cualquier dispositivo.

Panel administrativo (admin.html)

Centro de control: productos, pedidos, comisiones, vendedoras, bodegas, envíos, configuración.

Punto de venta (pos.html)

Pantalla táctil para cobrar en tienda física. Lee los mismos datos que la tienda en línea.

Login compartido (login.html)

Inicio de sesión único para todos los roles internos (admin, vendedor, bodega, agente_pedidos, superuser).

Reclutamiento (vendedoras.html)

Landing pública para que candidatas soliciten ser vendedoras. Los formularios llegan a Admin → Solicitudes.

Envíos (envios.html)

Vista de cotización de envíos Forza Delivery. También está embebida dentro de admin.html como pestaña.

Redireccionador (index.html)

Redirige automáticamente a Laurean.html. Punto de entrada cuando alguien visita la raíz del dominio.

Capas de la arquitectura

Backend: Supabase (Postgres en la nube)

Toda la data del negocio vive en Supabase: productos, pedidos, comisiones, perfiles de usuario, inventario. La base de datos es Postgres con RLS (Row Level Security) para que cada rol solo vea lo que le corresponde. El localStorage del navegador funciona únicamente como caché de catálogo con TTL de 5 minutos y como fallback offline, no como fuente de datos principal.

PWA — acceso sin conexion

El service worker (sw.js) cachea imágenes, CSS y fuentes por 7 días. La tienda puede mostrar el catálogo sin conexion usando el caché local. Requiere HTTPS (Vercel lo provee automáticamente).

Pila tecnológica

  • Frontend: HTML, CSS, JavaScript vanilla. Sin React, Vue ni bundler.
  • Backend: Supabase (Auth + Postgres + Storage + Edge Functions en Deno).
  • Deploy: Vercel. El "build" solo ejecuta node scripts/gen-config.js para generar js/config.js con las variables de entorno.
  • Estilos de marca: css/brand.css define los tokens de color (:root variables) compartidos entre todas las páginas.
  • Envíos: Forza Delivery, vía Edge Function forza-proxy que firma HMAC y protege la SecretKey.
Capítulo 02

Credenciales y matriz de accesos

El sistema usa Supabase Auth (email + contraseña). No hay cuentas seed precargadas en producción; el superusuario crea cada cuenta desde admin.html o desde el dashboard de Supabase.

Los cinco roles del sistema

superuser
Control total

Acceso completo incluyendo zona de peligro (reset de pedidos, comisiones, precios).

admin
Gestión completa

Productos, pedidos, vendedoras, configuración. No accede a zona de peligro.

vendedor
Venta + comisiones

Accede al POS (si tiene los flags activos) y ve su panel de comisiones en la tienda.

bodega
Precio mayorista

Ve catálogo con 30% de descuento. Para compras internas o al por mayor.

agente_pedidos
Toma de pedidos

Rol para agentes que gestionan pedidos sin acceso a administración completa.

PIN de descuento
Sin login

Permite aplicar descuento manual en checkout o POS sin cambiar de sesión. Configurado en Admin → Configuración.

Matriz de acceso por superficie

SuperficieRoles que pueden entrarQue ven / pueden hacer
login.html Todos los roles internos Punto de entrada compartido. Supabase Auth valida email + contraseña y devuelve un JWT con el rol.
admin.html admin, superuser Panel completo de gestión. El superuser adicionalmente ve la "zona de peligro" (acciones destructivas e irreversibles).
pos.html vendedor, admin, superuser, bodega — con flags activos Requiere que el perfil en Supabase tenga active = true y can_login_pos = true. Ver nota abajo.
Laurean.html Publico (sin login) + cualquier rol logueado Catálogo y checkout. Vendedoras ven su panel de comisiones al iniciar sesión.
vendedoras.html Publico (sin login) Landing de reclutamiento. Los formularios llegan a Admin → Solicitudes.

Si no puedes entrar al POS

La causa más común es que falte uno de los flags de perfil. Verifica en Supabase Dashboard → Table Editor → tabla profiles que tu cuenta tenga active = true, can_login_pos = true y un rol permitido (vendedor, admin, superuser o bodega). Si alguno de esos tres campos es false o null, el POS rechazará el ingreso aunque la contraseña sea correcta.

Como crear un usuario por rol

Opción 1 — desde admin.html (recomendado): Admin → Vendedoras/Usuarios → "Crear vendedora" o "Nuevo usuario". El sistema llama a la Edge Function create-user que crea la cuenta en Supabase Auth e inserta la fila en profiles con el rol correcto.

Opcion 2 — desde el dashboard de Supabase: Supabase Dashboard → Authentication → Add user. Luego insertar manualmente una fila en la tabla profiles con el id del usuario, el role correcto, active = true y can_login_pos según corresponda.

Restablecer contraseña: desde Supabase Dashboard → Authentication → buscar el usuario → "Send password reset" o desde Admin → Usuarios → Reset password.

PIN de descuento manual

Existe independientemente de los roles de usuario. Se configura en Admin → Configuración. Se recomienda cambiar el valor default el primer dia de operacion. Permite aplicar un porcentaje de descuento en checkout y POS sin necesidad de iniciar sesion con otra cuenta.

Capítulo 03

Cómo opera la tienda en línea

La tienda pública es la cara del negocio. Funciona en cualquier dispositivo (móvil, tablet, escritorio). El catálogo se carga desde Supabase; si no hay conexión, usa el caché local del navegador.

Navegación del catálogo

  • Cuatro categorías padre: Mujer, Hombre, Kids, Ofertas.
  • Cada una abre sus subcategorías (Blusas, T-Shirt, Pijamas, etc.).
  • Etiqueta "Nueva" automática en productos marcados como is_new_arrival.
  • Productos con flash_end entran a sección Ofertas con cuenta regresiva.

Carrito y checkout

  • Carrito flotante (drawer lateral) accesible desde cualquier pantalla.
  • Aumentar o reducir cantidad por producto.
  • Resumen de pedido antes de confirmar.
  • Cierre del pedido; el método de pago se coordina fuera de la plataforma por ahora (sin pasarela activa).
  • Cotización de envío Forza Delivery disponible en el checkout.

Multi-moneda

Cada producto puede tener precio en GTQ y USD. La tienda muestra el que corresponda según configuración.

Código referral de vendedora

  • El cliente puede ingresar un código tipo LAU-XXXX al hacer checkout.
  • Aplica descuento automático y registra la venta para comisión.
  • La vendedora ve su historial y comisiones en su panel personal dentro de la tienda.

Descuento manual con PIN

Para promociones puntuales o ventas especiales: ingresar el PIN configurado en Admin y aplicar el porcentaje en el checkout. Queda registrado en el pedido.

Panel del vendedor

Si una vendedora inicia sesión, ve un botón para abrir su panel: historial de ventas referidas, comisión pendiente, comisión pagada y su código referral con botón de copiar.

Capítulo 04

Cómo opera el panel administrativo

El admin es donde sucede toda la gestión. Acceso desde login.html con cuenta admin o superuser. Sidebar con todas las secciones del sistema.

Dashboard

KPIs principales del día: ventas, pedidos, ticket promedio, mes. Lista de pedidos recientes, usuarios activos y top vendedores/productos.

Pedidos

Listado completo con detalles. El superuser puede borrar en bloque desde la zona de peligro (para limpiar entornos de prueba o reiniciar operación).

Comisiones

Cálculo automático por venta cuando hay código referral. Listado por vendedora con totales pendientes y pagados.

Estadísticas

Filtros por rango de fechas, top 5 productos. Export CSV con un clic.

Productos

Crear, editar y ajustar precios en USD/GTQ. Asignar categoría padre, subcategoría y bandera de nuevo ingreso. Las imágenes se suben a Supabase Storage (bucket product-images).

Categorías

Crear nuevas categorías y gestionar precios por categoría completa.

Proveedores

Crear, editar y listar proveedores del negocio.

Bodegas

Gestión de inventario por bodega. La distribución de stock por bodega se está digitalizando (tablas inventory_items + inventory_stock).

Envíos

Cotizar tarifa Forza Delivery directamente desde el admin. Ver Capítulo 06.

Cotizaciones

Generar propuestas comerciales para clientes mayoristas con descuento global y total dinámico. Imprimible directo desde el navegador.

Vendedoras / Usuarios

Crear vendedoras, asignar código referral automático, ver listado. También gestiona los usuarios de todos los roles via Edge Function create-user.

Solicitudes

Bandeja de solicitudes pendientes (formularios enviados desde vendedoras.html).

Configuración

Solo superuser/admin. Ver Capítulo 09.

Zona de peligro

Borrar todos los pedidos · Borrar todas las comisiones · Restaurar precios base. Acciones irreversibles. Solo el superuser tiene acceso. Usar exclusivamente en entornos de prueba o cuando se quiera reiniciar la operación desde cero.

Capítulo 05

Cómo opera el POS

El Punto de Venta es la pantalla para cobrar en tienda física. Roles que pueden acceder: vendedor, admin, superuser, bodega — siempre que el perfil tenga active = true y can_login_pos = true.

Tres pestañas, un flujo

Venta

Catálogo en mosaico, búsqueda y filtro por categoría padre. Toque para agregar al carrito lateral. Nombre del cliente opcional. Botón "Cobrar" muestra el total.

Historial

Tabla con todas las ventas del día: hora, ID, cliente, items, total, descuento, método de pago y estado.

Cierre de caja

Cuatro KPIs (total ventas, descuentos otorgados, canceladas, ticket promedio) y Export CSV del día.

Métodos de pago

  • Efectivo
  • Tarjeta
  • QR

Se elige al momento de cobrar. Queda registrado para el cierre diario.

Descuento autorizado con PIN

Para aplicar un porcentaje (1% a 50%) sobre el subtotal: ingresar el PIN configurado en Admin → Configuración. El descuento queda visible en verde y registrado en el ticket.

Precios diferenciados por rol

Si la usuaria es vendedora ve precios con −15%. Si es bodega ve precios con −30%. Admin ve precios completos. Los porcentajes son configurables desde Admin → Configuración.

Capítulo 06

Envíos — integración Forza Delivery

Laurean Shop tiene integración nativa con Forza Delivery para cotizar tarifas de envío en tiempo real. La integración protege la SecretKey de Forza en el servidor (nunca expuesta en el navegador).

Como funciona la integración

  • El frontend llama al cliente js/forza-client.js, que a su vez invoca la Edge Function forza-proxy en Supabase.
  • La Edge Function firma el HMAC con la SecretKey de Forza (que vive como secret de Supabase, no en el código) y devuelve la tarifa.
  • Solo usuarios con rol admin o superuser pueden cotizar envíos (la Edge Function verifica el JWT).

Donde se usa

  • envios.html — página independiente para cotizar tarifas.
  • admin.html — pestaña de Envíos embebida en el panel.
  • Laurean.html — cotización de envío disponible durante el checkout.

Parámetros para cotizar

Se requiere código de origen (HeaderCodeSource), código de destino (HeaderCodeDestiny), peso y dimensiones del paquete. Forza devuelve las opciones de servicio disponibles con su tarifa en GTQ.

Secretos de Forza

La SecretKey y credenciales de Forza nunca van en el código fuente ni en el frontend. Viven exclusivamente como secrets de Edge Functions en Supabase (supabase secrets set). Si hay que rotar las credenciales, se hace desde el dashboard de Supabase sin tocar el código.

Capítulo 07

Logos e imágenes — cómo reemplazarlos

Los archivos de marca tienen nombres semánticos fijos en images/brand/. Para reemplazar un logo, basta subir el nuevo archivo con el mismo nombre: el sistema lo toma automáticamente en todas las páginas que lo usan.

Mapa de logos de marca

ArchivoDonde se usa
images/brand/favicon.svg Favicon (pestaña del navegador) en todas las páginas.
images/brand/loader.svg Pantalla de carga inicial (spinner). Aparece en Laurean.html, vendedoras.html, admin.html, login.html. Diseñado para fondo oscuro.
images/brand/logo-navbar.svg Logo del navbar y del login. Aparece en Laurean.html, pos.html, vendedoras.html. Diseñado para fondo claro.
images/brand/logo-oscuro.svg Logo sobre fondo oscuro. Aparece en el footer de Laurean.html, pos.html, admin.html, login.html.

Como reemplazar un logo

Para cambiar el logo del navbar, guarda tu archivo SVG con el nombre exacto logo-navbar.svg y súbelo a la carpeta images/brand/ reemplazando el existente. El sistema lo tomará automáticamente en todas las páginas que lo usan. Mismo principio para favicon.svg, loader.svg y logo-oscuro.svg. Luego hacer commit y push para que el deploy de Vercel tome el nuevo archivo.

Fotos de productos del catálogo

  • Las fotos de productos viven en Supabase Storage, bucket product-images.
  • Se suben desde Admin → Productos al crear o editar un producto (componente drag-and-drop con compresión automática).
  • También pueden estar en la carpeta local images/ como respaldo o referencia.
  • Recomendacion de normalizacion: para mantener armonía visual, las fotos del catálogo deben tener recorte, proporción y fondo consistentes (idealmente fondo neutro, formato 4:5 o 1:1). Fotos heterogéneas hacen que el catálogo se vea desprolijo.
Capítulo 08

Deploy y dominio (Vercel + GoDaddy)

El sitio se despliega en Vercel. El único paso de "build" es generar js/config.js con las variables de entorno de Supabase. No hay bundler ni transpilación.

Pasos de deploy en Vercel

  • Paso 1: Conectar el repositorio GitHub a Vercel (Add New → Project → Import).
  • Paso 2: Framework Preset: Other. El Build Command y Output ya vienen configurados en vercel.json (node scripts/gen-config.js, output .). No cambiar nada.
  • Paso 3: En Project → Settings → Environment Variables agregar para Production:
    • SUPABASE_URL = la URL de tu proyecto Supabase (https://<proyecto>.supabase.co).
    • SUPABASE_ANON = la anon/public key (Supabase → Settings → API). Esta clave es pública por diseño; RLS la protege.
  • Paso 4: Deploy. La raíz (/) redirige automáticamente a /Laurean.html.

Dominio GoDaddy

  • Vercel → Project → Settings → Domains → agregar tu dominio.
  • En GoDaddy → DNS Management configurar:
    • Dominio raíz (tudominio.com): registro A apuntando a 76.76.21.21.
    • Subdominio (www): registro CNAME apuntando a cname.vercel-dns.com.
  • Esperar propagación DNS (minutos a ~1 hora). Vercel emite el certificado HTTPS automáticamente.

Restringir CORS tras fijar el dominio

Una vez que el dominio está activo, ejecutar desde la CLI de Supabase:

supabase secrets set FRONTEND_ORIGIN=https://tu-dominio.com

Luego re-deployar las Edge Functions para que tomen el nuevo secret. Esto restringe las peticiones CORS solo al dominio de producción y permite que QPayPro acepte POSTs únicamente desde el sitio configurado.

Edge Functions requeridas

  • forza-proxy — cotización de envíos Forza Delivery.
  • forza-webhook — webhook de eventos de Forza.
  • create-user — creación de usuarios con rol desde admin.html.
  • log-visit — contador público de visitas del sitio.

Service Worker y HTTPS

El service worker (sw.js) requiere HTTPS para activarse. Vercel lo provee automáticamente. Al publicar una nueva versión de assets, actualizar la variable VERSION en sw.js para invalidar el caché viejo en los navegadores de los usuarios.

Mantenimiento y soporte

Qué cubre la mensualidad, cómo se piden cambios y cómo respondemos.

Lo que cubre la mensualidad

  • 4 a 5 cambios menores por mes (textos, imágenes, ajustes de catálogo, pequeños ajustes visuales, configuración).
  • Base de datos y respaldos (Supabase genera respaldos automáticos diarios).
  • Soporte por WhatsApp y correo en horario hábil.
  • Monitoreo y mantenimiento preventivo.
  • Actualizaciones de seguridad.

Cambios grandes

Funcionalidades nuevas, integraciones externas (pasarela de pago, factura electrónica, app nativa), rediseños completos o nuevas integraciones se gestionan aparte:

  • El cliente envía la solicitud.
  • Evaluamos viabilidad técnica y alcance.
  • Entregamos cotización separada.
  • El cliente aprueba antes de iniciar.

Solicitar respaldo de información

En horario hábil de mañana o tarde (días laborales), con tiempo de respuesta razonable. No se atienden solicitudes fuera de horario ni fines de semana.

Funciones en hoja de ruta (no disponibles hoy)
